# WatchTogether 需求 Todo

本文档用于把当前提出的房间同步、私有房免重复输密码、管理能力、空房间清理、跨房间加入行为等需求拆成可执行待办项。

关联仓库：

- Go 后端：`https://github.com/LKL1235/WatchTvTogether`
- Web 客户端：`https://github.com/LKL1235/WatchTvTogether-Web`

## 后端 todo（Go / Gin / Redis / Ably）

### 1. 新用户加入房间时同步播放状态

- [ ] 复现并确认当前行为：房主和房间内其他用户正在播放时，新用户通过 `POST /api/rooms/:roomId/join`、`POST /api/rooms/:roomId/snapshot` 进入后只同步到了视频/进度，但播放器仍停在暂停状态。
- [ ] 明确 Redis 房间状态结构中的权威字段：至少包含 `video_id`、`queue`、`playback_mode`、`action`（`play`/`pause`/`seek`/`switch`/`next`）、`position`、`updated_at`、`control_version`。
- [ ] 在房主提交 `POST /api/rooms/:roomId/control` 时，把播放/暂停动作和进度一起写入 Redis；播放状态为 `play` 时记录服务端时间戳，用于后续计算实时进度。
- [ ] 在 `POST /api/rooms/:roomId/snapshot` 返回时，如果房间状态是播放中，应返回 `state.action = "play"`，并返回按 `updated_at` 与当前时间修正后的 `state.position`，避免新用户拿到旧进度或默认暂停。
- [ ] 确认 `GET /api/rooms/:roomId/state` 与 `snapshot` 对播放状态的计算逻辑一致，避免一个接口返回播放中、另一个接口返回暂停。
- [ ] 对 Ably 广播消息 `room.sync`、`room.snapshot` 的 payload 做字段对齐，保证新加入用户初始化和房间内老用户实时同步使用同一套状态语义。
- [ ] 增加后端测试：覆盖暂停状态进入、播放状态进入、播放中经过一段时间后进入、切换视频后进入、无当前视频时进入等场景。
- [ ] 增加联调验收项：房主播放视频后，新用户加入房间应自动进入播放状态，且进度接近房主当前进度。

### 2. 私有房输入过密码后再次进入不应重复要求密码

- [ ] 设计服务端授权模型：用户通过私有房密码校验后，在服务端记录“用户已授权进入该房间”的状态，而不是依赖客户端长期保存明文密码。
- [ ] 确定授权记录存储位置与过期策略：优先使用 Redis，例如 `room:{roomId}:authorized_users` 或同类结构，记录 `userId`、授权时间、过期时间。
- [ ] 在 `POST /api/rooms/:roomId/join` 中实现逻辑：用户已授权、房主、站点管理员再次进入私有房时无需提交密码；首次进入或授权过期时仍需校验密码。
- [ ] 在 `POST /api/rooms/:roomId/snapshot` 中复用同一授权判断，避免用户 join 成功后进入房间页 snapshot 又被要求密码。
- [ ] 在 `POST /api/ably/token` 中复用同一授权判断，避免 Ably token 续签时又要求客户端传密码。
- [ ] 定义退出、被踢、房间关闭、房间密码变更时授权记录是否清理；建议被踢和密码变更应清理该用户/该房间授权。
- [ ] 明确接口错误码：未授权或密码错误返回 401/403，不再让客户端用 409 猜测是否需要弹密码框。
- [ ] 增加后端测试：首次无密码失败、密码正确后授权、再次 join 无需密码、snapshot 无需密码、Ably token 续签无需密码、密码错误不写授权、被踢后授权失效。

### 3. 管理功能加强：管理员手动关闭房间

- [ ] 梳理现有 `DELETE /api/rooms/:roomId` 行为：确认是否已存在关闭房间接口、权限校验、Redis 清理、数据库状态变更和 Ably 广播。
- [ ] 定义关闭房间权限：站点管理员可关闭任意房间，房主可关闭自己的房间，普通成员不可关闭。
- [ ] 定义关闭语义：关闭后房间不再出现在 `GET /api/rooms`，新用户不能 join，已在房间内用户收到房间关闭事件并退回大厅。
- [ ] 后端关闭房间时清理 Redis 中的房间状态、队列、成员、私有房授权记录、空房间检查时间戳等关联 key。
- [ ] 通过 Ably 发布 `room.event` 或 `room.snapshot`，事件建议包含 `event = "room_closed"`、`room_id`、`closed_by`、`reason`，让客户端可以展示提示并退出房间。
- [ ] 为管理员后台补充房间监控所需字段：当前用户数、房主、可见性、当前视频、创建时间、更新时间、是否空房间、是否播放中。
- [ ] 增加审计或日志：记录管理员/房主关闭房间的操作人、房间 ID、时间、结果。
- [ ] 增加后端测试：管理员关闭任意房间、房主关闭自己房间、普通用户关闭失败、关闭后不能加入、关闭后 Redis key 被清理、关闭事件被发布。

### 4. 服务端定时/触发式检查空房间清理

- [ ] 明确空房间判定来源：优先使用服务端维护的房间成员/Presence 映射；如使用 Ably Presence，需要有可靠的服务端侧成员缓存或定期同步策略。
- [ ] 在 Redis 中为每个房间维护“疑似空房间起始时间戳”，例如 `room:{roomId}:empty_since`；房间从非空变为空时写入当前时间。
- [ ] 当任意用户登录成功时触发一次轻量检查：扫描 Redis 中的空房间时间戳，如果 `now - empty_since >= 5 分钟`，则进入清理流程。
- [ ] 清理时再次确认房间内没有用户，避免扫描到旧时间戳后误删刚有人重新加入的房间。
- [ ] 清理没有用户的房间：删除房间 Redis 状态、队列、授权、成员缓存，并按产品规则决定是否软删除/关闭数据库房间记录。
- [ ] 房间重新有人加入时清理 `empty_since`，避免房间曾经为空导致后续误清理。
- [ ] 让手动关闭房间和空房间自动清理复用同一个内部 service，减少 Redis key 清理遗漏。
- [ ] 增加日志与指标：每次登录触发扫描的房间数量、实际清理数量、跳过原因、Redis 错误。
- [ ] 增加后端测试：登录触发扫描、未满 5 分钟不清理、满 5 分钟但有人重新加入不清理、满 5 分钟且无人则清理、多房间批量扫描。

### 5. 用户已在另一个房间时加入新房间应退出旧房间

- [ ] 复现并定位当前 409 来源：确认是用户-房间成员关系、Redis 当前房间记录，还是数据库唯一约束导致同一用户加入新房间返回 409。
- [ ] 定义目标行为：用户加入新房间时，服务端应自动从旧房间移除该用户，然后加入新房间；接口返回新房间信息，不再返回 409。
- [ ] 在 `POST /api/rooms/:roomId/join` 中加入事务/原子流程：查询用户当前房间，如果不同于目标房间，先执行 leave old room，再执行 join new room。
- [ ] 旧房间需要发布成员离开事件；如果用户是旧房间房主，需要定义房主转移或关闭房间规则。
- [ ] 如果旧房间因该用户离开变为空，应写入 `empty_since`，交给 5 分钟空房间清理流程处理。
- [ ] 如果用户加入的是同一房间，应幂等返回当前房间信息，不视为冲突。
- [ ] 确认 Ably token 和 Presence 的退出顺序：服务端状态先切换，客户端随后断开旧频道并连接新频道，避免同时占用两个房间。
- [ ] 增加后端测试：同房间重复 join 幂等、从公开房加入私有房、从私有房加入公开房、加入新房后旧房成员减少、旧房为空写入时间戳、旧房主切换房间的产品规则。

## 客户端 todo（WatchTvTogether-Web / Vue 3）

### 1. 新用户加入后按服务端播放状态初始化播放器

- [ ] 在房间页进入流程中以 `POST /api/rooms/:roomId/snapshot` 为权威来源，读取 `state.action`、`state.position`、`state.updated_at`、`state.video_id`、`queue`、`playback_mode`。
- [ ] 当 snapshot 返回 `state.action = "play"` 时，设置视频源与进度后主动调用播放器 `play()`；如果浏览器自动播放策略阻止播放，应显示“点击继续播放/解除静音后播放”的提示。
- [ ] 当 snapshot 返回暂停状态时，保持暂停并定位到服务端返回的进度。
- [ ] 对房主与普通成员使用同一套初始化逻辑；普通成员仍保持只读观看模式，但需要能被远端同步驱动播放。
- [ ] 加入状态应用防回环标记，避免初始化时触发原生 `play`/`pause`/`seeked` 后再次向服务端提交控制。
- [ ] 增加客户端测试或联调脚本：播放中加入、暂停中加入、切换视频后加入、刷新页面后恢复状态。

### 2. 私有房免重复输入密码的客户端配合

- [ ] 调整大厅加入私有房流程：先调用 `POST /api/rooms/:roomId/join` 不带密码尝试进入；仅在服务端返回明确“需要密码/未授权”时弹出密码框。
- [ ] 密码输入成功后不再长期保存明文密码；进入房间、snapshot、Ably token 续签应优先依赖服务端授权记录。
- [ ] 保留短期运行期密码兜底仅用于兼容后端未完成授权记录的阶段，并在后端能力就绪后移除。
- [ ] 处理授权过期或被踢后的 UX：如果 snapshot 或 Ably token 返回未授权，应退回大厅并提示需要重新输入密码或已被移出房间。
- [ ] 调整错误提示：密码错误、授权过期、房间已关闭、网络错误分别展示不同文案。
- [ ] 增加测试：首次进私有房需要密码、密码正确后再次进入不弹窗、刷新房间页不再要求密码、授权失效后重新弹窗。

### 3. 管理员手动关闭房间的客户端能力

- [ ] 在 `src/api.ts` 增加关闭房间 API 封装，例如 `DELETE /api/rooms/{roomId}`。
- [ ] 在管理员后台房间监控列表中增加“关闭房间”按钮，仅管理员可见；房主关闭自己房间可在房间页单独设计入口。
- [ ] 点击关闭前展示确认弹窗，说明房间内用户会被移出、播放状态和临时队列会被清理。
- [ ] 关闭成功后刷新房间列表和管理统计，并显示成功提示。
- [ ] 关闭失败时按 403/404/409/5xx 展示明确错误：无权限、房间不存在、状态冲突、服务异常。
- [ ] 房间页监听 Ably `room_closed` 事件：提示“房间已被管理员关闭”，断开实时连接，清理当前房间状态并返回大厅。
- [ ] 如果当前用户正在被关闭的房间内，避免继续请求 `state`、`control`、`ably/token` 导致错误刷屏。
- [ ] 增加测试：管理员关闭房间按钮、确认/取消流程、关闭后列表移除、房间内用户收到事件并返回大厅。

### 4. 空房间自动清理的客户端配合

- [ ] 大厅房间列表刷新时正确处理已被后端自动清理的房间：列表移除，不再保留本地旧房间卡片。
- [ ] 用户尝试进入已被清理房间时，针对 404/410 展示“房间已自动关闭或不存在”，并刷新房间列表。
- [ ] 管理员后台房间监控展示空房间相关信息：当前人数、空置开始时间、距离自动清理剩余时间（若后端提供）。
- [ ] 房间页实时成员为空/自己离开时不由客户端直接删除房间，只等待后端权威清理，避免误删。
- [ ] 增加测试：房间被自动清理后大厅刷新、管理员监控刷新、旧链接进入被清理房间的错误提示。

### 5. 加入新房间时自动退出旧房间的客户端配合

- [ ] 大厅点击加入新房间时，不再把服务端 409 当作最终失败；如果后端仍短期返回 409，应提示用户刷新或按兼容策略处理。
- [ ] 后端完成自动切房后，客户端收到新房间响应时应断开旧房间 Ably 连接、清理旧房间本地状态，再打开新房间。
- [ ] 如果旧房间收到自己的 leave/kick/room_closed 事件，而当前界面已经切到新房间，应忽略旧房间事件，避免误退回大厅。
- [ ] 管理当前 `currentRoom` 与 `currentJoinPassword`：切换房间时清空旧房间密码和旧队列/播放器状态。
- [ ] 加入同一房间时保持幂等：不重复打开页面、不重复连接 Ably、不重复弹密码框。
- [ ] 增加测试：从公开房切到公开房、从私有房切到公开房、从公开房切到已授权私有房、从旧房离开后旧房成员列表更新。

## 联调验收清单

- [ ] 房主播放视频后，新用户加入房间会自动播放，并定位到接近房主的当前进度。
- [ ] 房主暂停视频后，新用户加入房间会停在同一进度附近，不会自动播放。
- [ ] 普通用户首次输入私有房密码成功后，退出再进同一房间不再弹密码框。
- [ ] 私有房授权过期、被踢或房间密码变更后，再次进入需要重新授权。
- [ ] 管理员可以在后台手动关闭房间；房间内用户收到提示并返回大厅。
- [ ] 任意用户登录时触发空房间检查；超过 5 分钟无人房间被清理，未超过 5 分钟或重新有人加入的房间不被误清理。
- [ ] 用户已在房间 A 时加入房间 B，服务端自动退出 A 并加入 B，不返回 409。
- [ ] 房间切换后旧房间 Ably 连接关闭，新房间同步和成员列表正常。
- [ ] 大厅和管理员后台能正确处理房间被手动关闭或自动清理后的刷新、空态和错误提示。
