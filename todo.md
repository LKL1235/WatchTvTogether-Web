# WatchTogether 需求 Todo（Web 客户端镜像）

本文档与后端仓库 [PR #27 / todo.md](https://github.com/LKL1235/WatchTvTogether/pull/27) 中的客户端章节对齐；实现完成后将对应项勾选。

## 客户端 todo

- [x] 优化视频队列提交流程：用户点击添加视频时立即调用服务端队列接口提交新增项，并通过实时消息让房间内所有用户立刻看到最新队列，而不是等切换视频时才提交。
- [x] 在队列 UI 中展示服务端权威状态：客户端本地队列只能作为乐观更新或加载态，最终以服务端返回和 Ably 推送的队列顺序、当前播放项、播放模式为准。
- [x] 大厅添加刷新按钮：在房间列表页面提供手动刷新入口，点击后重新请求服务端房间列表，并处理加载中、请求失败、空列表和刷新后的房间状态变化。
- [x] 改为监听视频播放器原生操作：房主在播放器上执行播放、暂停、拖动进度、跳转等操作时，客户端应直接调用后端控制接口同步状态，不再依赖额外的播放/暂停/同步进度按钮。
- [x] 移除视频下方额外进度 UI：删除「进度」文字、额外进度条和与其绑定的手动同步交互，避免与播放器原生控制栏产生重复或状态不一致。
- [x] 将房主播放器操作同步给其他用户：房主触发播放、暂停或进度变化后，客户端调用后端接口，由服务端写入 Redis 并通过 Ably 广播；其他客户端收到消息后更新播放器状态。
- [x] 避免播放器事件回环：非房主客户端因收到同步消息而更新播放器时，需要区分远端同步与用户本地操作，避免再次调用控制接口造成循环广播。
- [x] 队列添加顺序播放模式：当前视频播放结束后，客户端根据服务端队列状态进入下一个视频；如果已经是最后一个视频，则按产品规则停止播放或停留在末尾。
- [x] 队列添加循环播放模式：支持循环播放当前队列；当前视频结束后自动进入下一个视频，最后一个视频结束后回到队列第一个视频。
- [x] 将播放模式同步到服务端：顺序播放/循环播放的切换应调用服务端接口保存到 Redis，并广播给房间内所有客户端，保证刷新和重新进入后仍保持一致。
- [x] 监听视频播放结束事件：播放器触发 ended 时，如果当前用户是房主，则根据服务端队列和播放模式请求切换到下一个视频；非房主只等待服务端/Ably 同步，不主动推进队列。
- [x] 禁用非房主播放器控制栏：非房主进入房间时播放器应处于只读观看状态，禁止播放、暂停、拖动进度、倍速、切换视频等会改变房间状态的控制行为。
- [x] 保留非房主必要观看能力：禁用控制栏时仍需评估是否允许全屏、音量、本地静音、画质等不影响房间共享状态的本地操作，并在 UI 上清晰区分可用与不可用控件。
- [x] 进入房间时恢复播放状态：客户端进入或刷新房间后调用服务端状态接口，使用服务端返回的当前视频、队列、播放模式、播放状态和计算后的进度初始化播放器。
- [x] 补充客户端联调测试：覆盖添加队列即时同步、大厅刷新、房主原生播放器控制同步、非房主禁止控制、顺序播放、循环播放、刷新后恢复进度和多用户同房间观看场景。（当前仓库已加入队列推进逻辑的单元测试 `npm run test`，完整 E2E 可在后续补充。）
